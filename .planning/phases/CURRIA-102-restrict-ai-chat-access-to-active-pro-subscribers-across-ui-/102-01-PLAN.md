<objective>
Restrict CurrIA chat to authenticated users with an active Pro subscription by adding one reusable server-side entitlement seam and applying it consistently across the chat page, chat APIs, and the main authenticated entry point.

Purpose: close the current authorization gap without rewriting billing, session ownership, or chat orchestration architecture.
Output: a strict Pro chat entitlement helper, protected chat routes, a server-rendered `/chat` block state, sidebar gating, and focused regression coverage.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> `/chat` page | Untrusted client navigation must not decide entitlement. |
| browser -> `/api/agent` and session chat routes | Callers can bypass hidden UI and hit endpoints directly. |
| app server -> `user_quotas` recurring metadata | Billing metadata is the only trust anchor for Pro chat entitlement in this phase. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-102-01 | E | `/api/agent`, `/api/session/[id]`, `/api/session/[id]/messages` | mitigate | Reuse one server-side entitlement helper and return `403` before chat payload creation, session creation, or async dispatch for non-Pro users. |
| T-102-02 | I | `/api/session/[id]`, `/api/session/[id]/messages` | mitigate | Preserve `getSession(params.id, appUser.id)` ownership resolution and layer Pro entitlement before serializing session or message data. |
| T-102-03 | S | `/chat` page and sidebar CTA | mitigate | Derive page and shell gating from server truth only; do not trust client props or hidden-button state for entitlement. |
| T-102-04 | T | recurring billing metadata interpretation | mitigate | Treat missing subscription id, non-Pro plan, inactive status, expired renewal date, and metadata read failures as denied-by-default. |
</threat_model>
