<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| export processor -> billing state machine | Render success and billing finalization can diverge under retries or partial failures |
| reconciliation routine -> financial adjustments | Automated repair must only run when evidence is sufficient |
| status routes -> authenticated clients | Stage and error detail must remain user-scoped and not leak cross-user billing data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-05 | T | `generate-billable-resume.ts` | mitigate | Use intent-keyed reservation reuse and terminal state checks so retries cannot create duplicate holds or duplicate finalization. |
| T-44-06 | R | reconciliation routine | mitigate | Require explicit evidence from job state, reservation state, and artifact metadata before releasing or finalizing a hold; otherwise mark for manual review. |
| T-44-07 | I | `/api/file/[sessionId]` and `/api/jobs/[jobId]` | mitigate | Preserve existing auth and ownership checks and expose only user-scoped stage or error details. |
| T-44-08 | D | processor failure paths | mitigate | Log and surface reservation, finalize, release, and reconciliation failures separately so stuck holds are detectable and repairable. |
</threat_model>
