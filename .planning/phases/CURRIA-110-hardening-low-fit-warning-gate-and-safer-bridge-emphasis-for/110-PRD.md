# Phase 110 PRD

## Title

Hardening: low-fit warning gate and safer bridge emphasis for job targeting

## Objective

Add two hardenings on top of the Phase 108 and 109 `job_targeting` stack:

1. Improve how the rewrite uses explicit evidence and cautious semantic bridges in partially adherent vacancies.
2. Add a low-fit warning gate so clearly off-target vacancies stop generating automatically and require explicit user confirmation before persistence, billing, and file generation.

The system must remain honest, domain-agnostic, and recoverable.

## Product framing

The user should no longer experience an off-target case as:

`O sistema gerou automaticamente uma versão que parece Java mesmo sem base real.`

They should experience it as:

`Esta vaga parece muito distante do seu currículo atual. Você pode gerar mesmo assim, mas recomendamos revisar.`

With CTA behavior:

- `Fechar`
- `Gerar mesmo assim (1 crédito)` when credits are available
- `Adicionar créditos` when credits are not available

## In scope

- Add a safe emphasis layer derived from `TargetEvidence` so explicit evidence is promoted more competitively without inventing unsupported experience.
- Keep bridges useful but cautious, especially for adjacent business contexts and adjacent tooling.
- Add a low-fit warning gate ahead of version persistence, billable generation, and credit reservation.
- Add a generic core-requirement coverage heuristic that distinguishes central requirements from peripheral similarities.
- Promote fit-related soft warnings into a recoverable blocking state when low-fit conditions are strong enough.
- Reuse the Phase 109 recoverable modal and override mechanics with new low-fit human copy.
- Add observability that explains why the low-fit gate fired.
- Add focused tests for off-target and partially adherent cases.

## Out of scope

- Do not hardcode Java-only, BI-only, RH-only, or domain-specific product rules.
- Do not relax anti-hallucination protections.
- Do not allow bridge claims to become direct skills when the resume does not prove them.
- Do not change ATS enhancement, no-target rewrite, highlight-only flows, or generic chat behavior.
- Do not add new billing flows outside the Phase 109 override path.
- Do not introduce a second targeting architecture outside the current `job_targeting` pipeline.

## Problem statement

Two failure modes must be corrected together:

### 1. Partially adherent vacancies are too conservative

In adjacent roles such as Sistemas/BI/RH, the system is staying safe but under-selling real evidence like:

- Power Automate
- APIs REST
- HTML
- CSS
- automação de fluxos
- integração de sistemas
- relatórios gerenciais
- dashboards

The rewrite should use those signals better while still refusing unsupported direct claims such as:

- RPA como skill direta
- People Analytics como experiência direta
- Power Apps
- FLUIG
- JavaScript
- sistemas de RH

### 2. Clearly off-target vacancies still generate automatically

For distant roles such as a strong Java backend vacancy, the system can already detect:

- `riskLevel = high`
- `matchScore` very low
- very high `unsupported_gap` share
- very low explicit evidence

But it still reaches automatic success because those signals remain soft warnings instead of becoming a recoverable block.

That is a product bug.

## Required outcomes

1. Partial-fit vacancies use explicit evidence and cautious bridges better.
2. Low-fit vacancies stop before automatic persistence, credit reservation, and file generation.
3. The user still has a recoverable path through explicit override.
4. Similarities at the edge of the role must not be enough to claim the target role directly.
5. Logs and trace must explain exactly why the system blocked automatic generation.

## Requirement 1: Safe targeting emphasis

Create an internal targeting-emphasis layer, derived from `TargetEvidence`, that drives rewrite emphasis without weakening skill-surface safety.

```ts
type SafeTargetingEmphasis = {
  safeDirectEmphasis: string[]
  cautiousBridgeEmphasis: Array<{
    jobSignal: string
    safeWording: string
    supportingTerms: string[]
    forbiddenWording: string[]
  }>
  forbiddenDirectClaims: string[]
}
```

Expected behavior:

- Explicit, normalized, and safely direct evidence should be easier for the rewrite to promote.
- Bridge-only signals may appear only through cautious wording.
- Skills remain a strong claim surface and must not accept bridge-only claims as direct skills.

Examples that should be allowed when explicitly supported:

- Power Automate
- APIs REST
- HTML
- CSS
- integração de dados
- automação de fluxos
- relatórios gerenciais

Examples that must remain forbidden as direct claims when unsupported:

- RPA
- People Analytics
- Power Apps
- Power Platform as broad mastery
- FLUIG
- JavaScript
- sistemas de RH

## Requirement 2: Low-fit warning gate

Add a new decision object before version persistence and before `generate_file`.

```ts
type LowFitWarningGate = {
  triggered: boolean
  reason:
    | "very_low_match_score"
    | "too_many_unsupported_core_requirements"
    | "target_role_not_supported"
    | "explicit_evidence_too_low"
    | "high_risk_off_target"
  matchScore: number
  riskLevel: string
  familyDistance?: string
  explicitEvidenceCount: number
  unsupportedGapCount: number
  unsupportedGapRatio: number
  explicitEvidenceRatio: number
  coreRequirementCoverage: {
    total: number
    supported: number
    unsupported: number
    unsupportedSignals: string[]
  }
}
```

The gate must trigger when any strong low-fit combination is true, such as:

- `riskLevel === "high"` with low `matchScore`
- high `unsupportedGapRatio` with low `explicitEvidenceRatio`
- many missing skills combined with a high unsupported ratio
- target-role positioning forbids direct role claim and explicit support remains weak
- core requirement coverage is too low

This gate must not permanently block the user.
It must only prevent automatic generation until explicit confirmation.

## Requirement 3: Core requirement coverage

Add a generic heuristic that separates:

- core requirements
- secondary requirements
- differential requirements

The system must identify central requirements from the vacancy itself instead of treating all matches equally.

Expected signals for `core`:

- appears in role title or target role
- appears in `Requisitos`, `Obrigatório`, or equivalent sections
- carries years-of-experience wording
- appears repeatedly
- reads like main stack / central capability
- appears in patterns such as `domínio`, `experiência com`, `profissional com`

Expected signals for `differential`:

- appears under `Diferenciais`, `Desejável`, or equivalent sections

Expected effect:

- peripheral matches like Git, SQL, APIs REST, and integrations may remain usable
- but they cannot mask missing core stack coverage in a clearly off-target vacancy

## Requirement 4: Promote fit-related warnings into recoverable block

When `lowFitWarningGate.triggered === true`, fit-related warnings must escalate into a recoverable blocking state if they indicate overclaim or unsupported presentation.

Expected promoted issue families:

- `target_role_overclaim`
- `summary_skill_without_evidence`
- `unsupported_claim`

When promoted:

- `validation.blocked = true`
- `validation.recoverable = true`
- no automatic final persistence
- no automatic credit reservation
- no automatic file generation
- the recoverable block/modal path must be used

## Requirement 5: Human low-fit modal

When the low-fit gate triggers, the user-facing modal must explain:

- the vacancy appears far from the current resume
- the system found too few proven points for the main requirements
- there may be adjacent evidence, but it does not justify direct self-presentation as the target role
- the user may still generate and revise manually

Copy constraints:

- no internal words such as `unsupported_gap`, `targetEvidence`, `pipeline`, `hardIssue`
- PT-BR UTF-8 only
- reuse the Phase 109 modal shell and credit-aware CTA logic

## Requirement 6: No automatic billing or file generation under low-fit

If the low-fit gate triggers:

- do not call `generate_file` automatically
- do not reserve credit automatically
- do not persist the version as final automatic success
- do not complete the flow silently

Only after explicit override confirmation should the existing Phase 109 path:

- persist the version
- create the CV version
- call `generate_file`
- consume one credit on success
- save `validationOverride`

## Requirement 7: Observability

Add trace and logging data that explain:

- whether the low-fit gate was evaluated
- whether it triggered
- the reason
- `matchScore`
- `riskLevel`
- `familyDistance`
- `explicitEvidenceCount`
- `unsupportedGapCount`
- `unsupportedGapRatio`
- `explicitEvidenceRatio`
- `coreRequirementCoverage`
- which warnings were promoted from soft to recoverable-hard

Expected trace additions:

```ts
trace.lowFitGate = {
  evaluated: true,
  triggered: boolean,
  reason?: string,
  matchScore,
  riskLevel,
  familyDistance,
  explicitEvidenceCount,
  unsupportedGapCount,
  unsupportedGapRatio,
  explicitEvidenceRatio,
  coreRequirementCoverage
}

trace.validation.promotedWarnings?: Array<{
  from: "soft"
  to: "recoverable_hard"
  issueType: string
  reason: string
}>
```

And `agent.job_targeting.plan_built` should expose:

- `targetRolePositioning.permission`
- `unsupportedGapRatio`
- `explicitEvidenceRatio`
- `lowFitGateTriggered`

## Requirement 8: Off-target role integrity

Even when peripheral similarities exist, the system must still forbid unsupported core-role claims.

For a distant backend Java vacancy, examples that may be claimable if the resume proves them:

- Git
- APIs REST
- SQL / bancos relacionais
- integrações

Examples that must remain forbidden without real evidence:

- Java
- 5+ anos de Java
- Desenvolvedor Java
- Spring Boot
- JPA / Hibernate
- Kafka / RabbitMQ
- microsserviços
- Docker
- CI/CD
- SOLID
- orientação a objetos
- testes automatizados
- Kubernetes
- observabilidade
- mentoria da stack alvo

## Mandatory test scenarios

### Test A: Java off-target triggers low-fit warning

Expected:

- `lowFitGate.triggered = true`
- recoverable validation block returned
- no automatic file generation
- no automatic credit reservation

### Test B: Soft warnings are promoted under low-fit

Expected:

- promoted warnings recorded
- blocked recoverable validation returned

### Test C: Forbidden target skills stay out of skills surface

Expected:

- unsupported core stack like Java/Spring/Kafka/Docker/CI-CD does not appear in `skills` without evidence

### Test D: Peripheral similarities do not unlock the target role

Expected:

- Git / SQL / APIs REST may remain claimable
- but the role itself is still not directly claimable
- low-fit gate still triggers

### Test E: Low-fit modal uses human framing

Expected:

- explains the vacancy is distant
- explains what the resume proves better
- does not contain internal implementation vocabulary

### Test F: Override still works with credit

Expected:

- CTA `Gerar mesmo assim (1 crédito)`
- existing override path persists, generates, charges once, records override metadata

### Test G: No credit opens pricing

Expected:

- CTA becomes `Adicionar créditos`
- pricing flow opens
- no override call is made
- draft/token remain usable after credit refresh

### Test H: Partially adherent vacancy does not block when safe

Expected:

- explicit evidence is emphasized better
- bridge-only claims remain cautious
- forbidden unsupported direct claims stay forbidden

## Acceptance criteria

1. Off-target vacancies no longer auto-generate final artifacts silently.
2. Users can still explicitly generate through the existing recoverable override path.
3. Partial-fit vacancies become more competitive using real evidence already in the resume.
4. Forbidden unsupported claims remain forbidden.
5. Similarities at the edge of the role do not falsely unlock direct target-role presentation.
6. The trace explains exactly why the low-fit gate fired and why warnings were promoted.

## Definition of done

- Safe targeting emphasis or equivalent structure implemented.
- Rewrite promotes explicit evidence better in partial-fit cases.
- Rewrite uses cautious bridges without turning them into direct skill claims.
- Low-fit warning gate implemented.
- Core requirement coverage heuristic implemented.
- Soft warnings are promoted into recoverable block under low-fit.
- Off-target cases stop before automatic persistence, billing, and file generation.
- Low-fit modal reuses the Phase 109 credit-aware CTA behavior.
- Override path still works with credit and pricing path still works without credit.
- Trace includes low-fit gate evaluation and warning promotion details.
- Targeted tests for off-target, partial-fit, forbidden skills, and cautious bridges are added.
- `npm run typecheck` passes.
- Phase 108 and 109 regression suites continue to pass.
