## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> `POST /api/profile/smart-generation` | input de vaga e request de geração são não confiáveis |
| browser -> `POST /api/session/[id]/job-targeting/override` | token de override, sessionId e intenção de cobrança vêm do cliente |
| `job_targeting` pipeline -> persisted `agentState` | draft bloqueado e metadados de override precisam ficar íntegros e scoped ao usuário |
| override route -> billable generation | persistência de versão e cobrança precisam permanecer idempotentes e ordenadas |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-109-01 | T/E | `POST /api/session/[id]/job-targeting/override` | mitigate | Validar ownership, workflowMode, token opaco, expiração, fingerprint do draft e invalidar/reconsumir o token após uso para bloquear replay e override cross-session. |
| T-109-02 | T/I | `job_targeting` prompt + validation | mitigate | Aplicar `targetRolePositioning` per D-04 e revalidar claims diretos no resumo/skills/experience para impedir overclaim factual em roles distantes. |
| T-109-03 | R | override audit trail | mitigate | Persistir `validationOverride`, issues e contexto de cobrança em `agentState` e `credit_reservations.metadata` com idempotency key de override, mantendo logs nomeados por evento. |
| T-109-04 | D | billing path no override | mitigate | Reusar `generate_file`/reservation state machine, cobrar só após persistência + artifact success e falhar fechado quando draft/token estiver inválido ou faltando. |
| T-109-05 | I | ATS/non-target surfaces | mitigate | Guardar todos os novos campos e respostas atrás de `workflowMode === "job_targeting"` e `recoverable === true`, sem modal/token em falha técnica nem ATS. |
</threat_model>
