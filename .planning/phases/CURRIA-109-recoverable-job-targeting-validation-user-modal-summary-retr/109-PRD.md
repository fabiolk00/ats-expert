# Phase 109 PRD

## Title

Recoverable job targeting validation: user modal, summary retry and paid override

## Objective

Transform `job_targeting` factual validation failures into a recoverable product experience:

1. Try to automatically repair the summary when the failure is localized there.
2. If the rewrite still fails, show a clear and human validation modal.
3. Explain the real issue in user language.
4. Let the user close the modal or proceed with `Gerar mesmo assim (1 crédito)`.
5. Persist the generated target version with explicit validation override metadata when the user confirms.
6. Keep anti-hallucination protections as the default behavior.

## Product framing

The user must not experience this as:

`Você não pode gerar esse currículo.`

They should experience it as:

`Geramos uma versão, mas ela pode exagerar alguns pontos. Você pode revisar ou gerar mesmo assim.`

## In scope

- Add target-role positioning policy to the enriched targeting plan.
- Prevent direct self-presentation as the target role when the career fit is distant or evidence is weak.
- Add automatic retry for the summary only when validation failure is localized and recoverable.
- Enrich validation issues with actionable metadata such as `offendingSignal`, `offendingText`, and `suggestedReplacement`.
- Create a user-facing recoverable validation modal with human copy.
- Add the CTA `Gerar mesmo assim (1 crédito)`.
- Allow controlled persistence of the targeted version after user confirmation.
- Record validation override details in trace, metadata, and analytics.
- Ensure the override CTA only appears for recoverable validation blocks, never for technical failures.
- Fix mojibake and encoding regressions in user-facing strings.

## Out of scope

- Do not remove validation.
- Do not relax anti-hallucination rules by default.
- Do not convert `unsupported_gap` into an allowed direct claim.
- Do not change generic ATS enhancement behavior.
- Do not change highlight-only flows without target rewrite.
- Do not change generic rewrite behavior.
- Do not add domain-specific hardcode for BI, RH, dados, or tecnologia.
- Do not alter the core semantic taxonomy introduced in Phase 108.

## Expected flow

1. User requests job targeting for a vacancy.
2. System generates the targeted rewrite.
3. Validation finds a factual issue.
4. If the issue is summary-only and recoverable, the system retries only the summary with stricter instructions.
5. If revalidation passes, return the resume normally.
6. If it still fails, return a user-facing modal plus an override token/draft.
7. User can close or confirm `Gerar mesmo assim (1 crédito)`.
8. If the user confirms and persistence succeeds, charge exactly one credit, persist the CV version, mark validation override metadata, and show the generated resume.

## Requirements

### 1. Target role positioning

Add a new enriched plan field:

```ts
type TargetRoleClaimPermission =
  | "can_claim_target_role"
  | "can_bridge_to_target_role"
  | "must_not_claim_target_role";

type TargetRolePositioning = {
  targetRole: string;
  permission: TargetRoleClaimPermission;
  reason: string;
  safeRolePositioning: string;
  forbiddenRoleClaims: string[];
};
```

Expected rule:

- If `careerFit.riskLevel === "high"`, or `careerFit.familyDistance === "distant"`, or `unsupportedGapRatio >= 0.5`, then `permission = "must_not_claim_target_role"`.

### 2. Rewrite prompt must respect safe positioning

When `permission = "must_not_claim_target_role"`, the targeted rewrite prompt must forbid direct self-presentation as the target role and instruct the model to use `safeRolePositioning` instead.

### 3. Bridge classification should stay honest for distant roles

Before classifying a target signal as `unsupported_gap`, the system should still allow generic semantic bridges when the resume proves:

- the central activity but not the exact business domain;
- the family/category but not the exact tool;
- an adjacent practice but not the vacancy label;
- automação, integração, análise, governança, operação, atendimento, liderança, or melhoria without the exact business context.

These bridges must never become direct claims.

### 4. Automatic summary retry

Retry only when all of the following are true:

- `validation.blocked === true`;
- hard issues are localized to `summary`;
- issue type is one of:
  - `unsupported_claim`
  - `target_role_overclaim`
  - `summary_skill_without_evidence`

Do not retry when:

- issues affect multiple critical sections;
- skills contain forbidden claims;
- experience invents unsupported responsibilities;
- the error is technical or infra-related;
- there is no usable optimized draft.

Trace additions:

```ts
rewrite.summaryRetryAttempted: boolean;
rewrite.summaryRetrySucceeded: boolean;
rewrite.summaryRetryReason?: string;
```

### 5. Validation issues must carry user-facing context

Add or populate these optional fields:

```ts
type RewriteValidationIssue = {
  section: string;
  message: string;
  severity: "soft" | "hard";
  issueType?:
    | "unsupported_claim"
    | "unsupported_skill"
    | "target_role_overclaim"
    | "seniority_inflation"
    | "ungrounded_bridge"
    | "forbidden_claim";
  offendingSignal?: string;
  offendingText?: string;
  evidenceLevel?: EvidenceLevel;
  rewritePermission?: RewritePermission;
  suggestedReplacement?: string;
  userFacingTitle?: string;
  userFacingExplanation?: string;
};
```

### 6. Recoverable validation modal

Create:

```ts
buildUserFacingValidationBlockModal(args: {
  targetRole?: string;
  validationIssues: RewriteValidationIssue[];
  targetEvidence?: TargetEvidence[];
  originalProfileLabel?: string;
}): UserFacingValidationModalPayload
```

The modal must:

- explain the real problem in plain language;
- avoid internal terms such as `validation_failed`, `hardIssue`, `softWarning`, `unsupported_gap`, `targetEvidence`, `rewritePermission`, and `pipeline`;
- always show `Fechar`;
- show `Gerar mesmo assim (1 crédito)` only for recoverable validation blocks.

### 7. Paid override behavior

When the user confirms override:

1. validate the override token/draft ownership and expiry;
2. confirm the user has credit available;
3. charge one credit only if persistence succeeds;
4. persist the targeted CV version even with failed validation;
5. save validation issues and `validationOverride` metadata;
6. return the generated resume to the UI.

Do not charge if:

- the user closes;
- persistence fails;
- there is no draft to save;
- the error is technical.

### 8. Pending draft / override token

Store a temporary blocked targeted rewrite draft with:

```ts
type BlockedTargetedRewriteDraft = {
  id: string;
  sessionId: string;
  userId: string;
  optimizedCvState: CVState;
  originalCvState: CVState;
  targetJobDescription: string;
  targetRole?: string;
  validationIssues: RewriteValidationIssue[];
  expiresAt: string;
};
```

Expected API outcome:

```json
{
  "status": "validation_blocked_recoverable",
  "modal": {},
  "overrideToken": "..."
}
```

Expected override endpoint:

`POST /api/session/:id/job-targeting/override`

### 9. CTA gating rules

Show `Gerar mesmo assim (1 crédito)` only when:

- the block is factual validation, not technical failure;
- `optimizedChanged === true`;
- a usable optimized draft exists;
- persistence has not already failed;
- the user is allowed to generate.

For technical failures, show a generic retry message instead and no credit CTA.

## Mandatory tests

- Target role distant case must not directly claim the target role.
- Summary-only recoverable validation must trigger summary retry.
- Skills/experience violations must skip summary retry and go directly to modal.
- Modal text must avoid technical debug vocabulary.
- Modal must surface the real offending signal and suggested replacement.
- Override CTA must appear only for recoverable validation blocks.
- Override CTA must not appear for technical errors.
- Override success must charge exactly one credit and persist the CV version with override metadata.
- Override persistence failure must not charge credit.
- User-facing strings must preserve UTF-8 accents.
- Existing Phase 108 targeted rewrite protections must continue to pass.

## Acceptance criteria

- Recoverable factual validation failures no longer end as a dead-end blocker.
- The user sees a human explanation and a clear choice.
- Summary-only failures are retried automatically once before showing the modal.
- Distant target roles cannot become direct professional identity claims.
- Semantic bridges remain cautious and honest.
- Credit is charged only on successful override persistence.
- Validation override metadata and issues are persisted for audit.
- ATS and non-target flows remain behaviorally unchanged.

## Validation gate

- Phase 108 existing relevant suites remain green.
- New targeted tests for retry, modal, override, credit, and encoding pass.
- `npm run typecheck` passes.
