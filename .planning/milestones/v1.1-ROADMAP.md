# Roadmap: CurrIA

## Overview

This roadmap starts milestone `v1.1 Agent Reliability and Response Continuity` after the `v1.0` launch-hardening baseline shipped. The new work focuses on a live agent reliability incident: short dialog follow-ups like `reescreva` can still truncate, degrade, and surface repeated vacancy bootstrap copy. The phases therefore prove live `/api/agent` parity first, then harden dialog recovery and model routing, and finally verify the exact transcript the user sees.

### Phase 9: ATS Enhancement Reliability Hardening

**Goal**: Harden the deterministic ATS-enhancement pipeline so it remains section-consistent, fact-safe, operationally observable, and affordable under real resume workloads.
**Requirements**: [REL-01, REL-02, REL-03]
**Depends on**: Phase 8
**Success Criteria** (what must be TRUE):
  1. Sending only a resume triggers `ats_enhancement`, produces `atsAnalysis`, runs the full rewrite, returns `optimizedCvState`, validates against invented facts, persists a new optimized version, and drives preview/export from that optimized snapshot.
  2. The rewrite path is resilient to large payloads and section drift by using section-based execution, automatic validation, and step-level retries instead of one fragile monolithic rewrite.
  3. ATS-enhancement runs emit structured workflow logs that make phase-by-phase cost, failure points, and consistency issues observable.
**Risks**:
  - rewrite too large for a single LLM call
  - low consistency between rewritten sections
  - hallucinated bullets or unsupported claims
  - high cost from multiple LLM calls
**Mitigations**:
  - rewrite by section
  - automatic factual validation
  - retry by step
  - structured logs by phase
**Plans**: 3 plans

Plans:
- [x] 09-01: Add stage-aware ATS execution state and section-level retry handling
- [x] 09-02: Strengthen factual and cross-section consistency validation
- [x] 09-03: Add structured observability and protect last-valid optimized export behavior

### Phase 12: OpenAI Resilience and Async Processing Hardening

**Goal**: Prevent provider degradation or heavy PDF workloads from turning into cascading timeout failures across resume-generation flows.
**Requirements**: [OPS-08, OPS-09]
**Depends on**: Phase 11
**Success Criteria** (what must be TRUE):
  1. OpenAI-backed resume routes enforce bounded timeout budgets, controlled retries, and a circuit-breaker path that degrades predictably instead of stacking long provider waits.
  2. Resume and vacancy processing paths emit enough observability to show breaker opens, timeout causes, and fallback frequency during incidents.
  3. PDF extraction no longer relies on an unbounded synchronous route path for larger or slower inputs; heavy work is queued, staged, or otherwise decoupled from request timeout limits.
**Risks**:
  - circuit-breaker thresholds can be too aggressive and block healthy traffic
  - async PDF staging can break current upload UX if state transitions are unclear
  - multiple timeout layers can conflict and hide the real failure source
**Mitigations**:
  - per-provider and per-route breaker metrics with explicit open and half-open logs
  - staged rollout with fixture coverage for normal, slow, and failing provider behavior
  - queue or staged-processing UX that exposes pending and failed states explicitly
**Plans**: 3 plans

Plans:
- [x] 12-01: Add OpenAI timeout budgets, bounded retries, and circuit-breaker state
- [x] 12-02: Decouple heavy PDF processing from synchronous request execution
- [x] 12-03: Add observability and regression coverage for degraded provider and PDF paths
